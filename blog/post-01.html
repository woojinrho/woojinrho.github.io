The abstraction gap

Modern software engineering is all about abstractions. Frameworks hide databases, databases hide filesystems, filesystems hide disk controllers. Most of the time, that's great — it lets you move fast. But abstractions have a cost: you stop understanding what's actually happening.

Building a processor forces you to close the abstraction gap. When you write lw x5, 0(x10) and trace what actually happens — the fetch, decode, address calculation, memory read, writeback — you never look at memory access the same way again.

Debugging as a skill

Verilog is humbling. A waveform viewer showing five simultaneous pipeline stages, a stale value propagating through a forwarding mux, a hazard you forgot to detect — debugging hardware is a different skill from debugging software.

You can't put a print statement in hardware. You have to think before you simulate.

That discipline — thinking through state transitions before touching the keyboard — transfers everywhere. I'm slower to write code now. And faster to get it right.

The joy of it

Honestly, there's also just something deeply satisfying about it. The moment a simulated processor successfully runs a Fibonacci sequence you wrote in assembly, using a datapath you designed from scratch — that feeling doesn't go away.

Build the thing. All the way down.
