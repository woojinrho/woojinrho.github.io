RISC-V RV32I Processor

A complete implementation of the RISC-V RV32I base integer ISA in Verilog, built as a three-phase portfolio project: single-cycle, pipelined, and hardware-accelerated.

Cover image / block diagram

Overview

The RISC-V RV32I Processor project is a ground-up hardware implementation of the RISC-V base integer instruction set. Rather than following a textbook step-by-step, the goal was to build a production-quality RTL design with clean module boundaries, complete test coverage, and extensibility in mind.

The project is split into three phases, each building on the last:

  • Phase 1 — Single-cycle core (all RV32I base instructions)
  • Phase 2 — 5-stage pipeline with hazard detection and forwarding
  • Phase 3 — Matrix-multiply hardware accelerator via MMIO

Phase 1 — Single-Cycle Core

The single-cycle implementation processes one instruction per clock cycle. All eight modules — the instruction memory, register file, ALU, control unit, immediate generator, data memory, branch comparator, and top-level datapath — were written and verified independently before integration.

Module structure

riscv_top.v         // Top-level datapath
├── imem.v          // Instruction memory (synchronous ROM)
├── regfile.v       // 32×32-bit register file
├── alu.v           // Arithmetic/logic unit
├── control.v       // Opcode decoder / control signals
├── immgen.v        // Immediate value generator
├── dmem.v          // Data memory (byte-addressable)
├── branch_comp.v   // Branch comparator
└── mux2.v          // Generic 2:1 mux (parameterised width)

Key design decisions

The register file uses two asynchronous read ports and one synchronous write port, matching the semantics expected by the RV32I ISA. The ALU supports all required operations: ADD, SUB, AND, OR, XOR, SLT, SLTU, SLL, SRL, SRA.

Why implement a branch comparator separately instead of reusing the ALU? Because it allows the comparator to be optimised for equality/signed-less-than checks without polluting the ALU's critical path.

Phase 2 — Pipelined Core

The 5-stage pipeline (IF → ID → EX → MEM → WB) introduces hazards that require both detection logic and forwarding paths. Three categories of hazards are handled:

  • Data hazards — forwarding from EX/MEM and MEM/WB stages
  • Load-use hazards — one-cycle stall inserted by the hazard unit
  • Control hazards — branch resolved in MEM, two-cycle flush on taken branch

Phase 3 — Matrix Multiply Accelerator

The accelerator offloads matrix multiplication to a dedicated hardware unit, accessible to software via memory-mapped I/O registers. The CPU writes operand addresses and dimensions to control registers, triggers execution, and polls a status register for completion.

Results & Takeaways

All RV32I instructions pass the official RISC-V compliance tests. The pipelined core achieves a CPI of approximately 1.05 on typical integer benchmarks (accounting for stalls). The accelerator demonstrates a 12× speedup on 64×64 matrix multiplication versus pure software on the same core.

The biggest lesson: clean module boundaries and a consistent naming convention make debugging Verilog dramatically easier. Waveform tracing through five pipeline stages with cryptic signal names is an experience worth avoiding.
